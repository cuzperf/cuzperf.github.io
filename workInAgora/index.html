类型 格式

其他任意指针类型 p
enum（建议打印时用转一下） static_cast u

不知道的时候可以强转一下（总结一下各种转化）

由于 -Wformat-signedness 是 gcc 独有的，而我们使用的是 msvc 和 clang，所以目前 format 不区分 signed 和 unsigned，即PRIu
64 和 PRId64 这种混淆使用是无法检测出来的。
PRIu64 等这些宏都是 string-literal 即字符串常量（所以可以编译期拼接），定义在 inttypes.h 中，然后各个平台的定义会有一些出
入，从而做到跨平台。agora_log_i.h 已经#include ，所以不需要额外包含头文件
C++11requiresaspacebetweenliteralandidentifier[-Wreserved-user-defined-literal]，即上述宏旁边有字符串的话，需要加空格
其实对于 uint32_t，uint16_t, uint8_t 这种也应该分别使用PRIu32, PRIu16, PRIu8，但是考虑各个平台的定义都是一样的，并且由于
整形提升，所以用 %u 也没有啥问题

经验

  • windows arm 的编译：x64 的 VS2019 可以交叉编译 arm64（toolchain --add Microsoft.VisualStudio.Component.VC.Tools.ARM --add Microsoft.VisualStudio.Component.VC.Tools.ARM64），VS2022 可以直接在 win11 arm 上直接编译 arm64。目前 windows 必然定义 _WIN32，然后 cpu 相关的宏有三个：_M_IX86, _M_IX64 和 _M_ARM64（注意很多头文件在此 arch 上会报错）。ffmpeg 等需要 clang-cl 编译，并且需要 clang-cl.exe --target=arm64-windows （否则一些汇编代码无法识别），对于 MSVC 则需要 --target=aarch64-pc-windows-msvc 或者 --target=arm64-pc-windows-msvc。目前 windows arm 下还有很多指令并不支持，依然需要做适配的工作。
  • 是否应该把多明确一些 assumption（因为假设会随着时间可能不再成立）

Trace 的原理和解读
当前涉及到 trace 的模块
media_engine2/third_party/trace/ 中有两个 gn 文件：agora_trace.gni, BUILD.gn 以及 trace_record.h 和 trace_record.cc（尽可
能的使用纯 C），以及
rte_sdk/cpp/api2/internal/ 中agora_service_i.h 即开启 Trace 和关闭 Trace 的接口（作为 lowLevel API）
rte_sdk/src/main/core/ 中有 agora_service_impl.h 和agora_service_impl.cpp 中调用下面的diagnostic_service_impl 中的实现
rte_sdk/src/main/core/ 中有diagnostic_service_impl.h 和diagnostic_service_impl.cpp 里面有控制 Trace 的实现，且在
StopTrace 将 trace.log 时候写入到文件
现在 trace 已经做成 highlevel api，通过 setParameters 的方式调用 (2021.12.22 )
trace 的使用
如果设置了 FEATURE_ENABLE_TRACE=false，那么所有 Trace 的部分都会失效，且无任何副作用，不占用 CPU
trace 最多可以附带两个参数，一般认为是足够了，如果再多会对内存要求太高（且需要改内部代码，因为是一个结构体）
我们一般使用 AG_TRACE_FUN 的宏放在放在开头使用，带参数就使用AG_TRACE_FUN1，AG_TRACE_FUN2。AG_TRACE_EVENT 跟AG_TRACE_FUN
基本一致，只是更加灵活一点。它们最终都是调用 AG_TRACE_UNIQUE_OBJ，这个宏会创建一个对象，创建即开始记录，析构时结束
